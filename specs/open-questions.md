**Exclusive vs inclusive range bounds ([ADR-0005](adr/0005-reference-range-frameworks.md), [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §3/§5)**
`framework_ranges` bounds are **inclusive** on both ends ([ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §3: `T = [L, H]`), but real clinical cutoffs are frequently **exclusive**: MedlinePlus desirable cholesterol is *"less than 200"* (200 is borderline-high, not desirable), ADA prediabetes *begins at* A1C 5.7 and FPG 100, AHA/CDC low-risk hs-CRP is *"<1.0"*. The current seed works around the mismatch two different ways, and the inconsistency is real (CodeRabbit, PR #28): continuous measures store the threshold itself (200/100/150/1.0), which is the best inclusive approximation of "< threshold" but classifies the exact threshold value as `in_range`; glucose/A1c store threshold-minus-a-unit (99/5.6), correct for their reporting granularity but wrong for a fractional value in the gap. **Subtracting a display unit is explicitly not the fix** — it buries an unstated precision assumption in the data and breaks for any lab reporting finer precision.

**Owner's design steer (2026-07-17), to drive the range-model PR:** the comparison engine should use **standard mathematical interval notation with per-bound closure** — `(1, 10]` is not `[1, 10]` — rather than assuming both ends inclusive. And the inclusive/exclusive choice per bound should be a **user-verified layer**, not silently baked into a magnitude: the stored range carries its own closure (a per-bound open/closed flag, or a comparator per bound), the owner confirms it, and cited sources inform which end is open. This composes with — and is a strict generalization of — [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §3's existing open/closed handling on the *result* interval (a censored `<0.5` is already `(-inf, 0.5)`); today only the *target* interval is forced closed. Mechanism candidates: `range_low_closed`/`range_high_closed` boolean columns (default true = today's behaviour, so additive and back-compatible), or a comparator per bound mirroring the ADR-0030 result-value comparator domain. This is the same **range-model-expressiveness** theme as the cohort-dimension and `physical_min` entries below/above and should very likely land in one range-model PR with them; the comparison core (`ranges.py`) already has the open/closed interval machinery, so most of the change is schema + seed + the resolution read. Trigger: the range-model PR, or the first time a value landing exactly on a stored threshold is flagged visibly wrong to the owner.

**Reference ranges that depend on more than the biomarker — sex, age, fasting state ([ADR-0005](adr/0005-reference-range-frameworks.md), [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §5)**
`framework_ranges` is keyed by `(framework_id, biomarker_id, effective_date)`. It can therefore hold **one target per biomarker per framework per date** — and real reference data routinely needs several, differing along dimensions the table cannot express. Three concrete instances surfaced in Phase 3 WI-3, all in a seed of only six rows (seven counting the HDL range left unseeded over exactly this gap):

- **Sex.** NIH MedlinePlus states HDL's low cutoff as `< 40 mg/dL` (men) and `< 50 mg/dL` (women). Not representable; picking one silently is not acceptable.
- **Age.** The same source states different lipid targets for age ≤ 19 (TC < 170, LDL < 110) than for 20+ (TC < 200, LDL < 100). The adult values are seeded — correct for this platform's single adult owner, but an assumption the schema cannot state.
- **Fasting state.** The ADA glucose thresholds are *fasting* plasma glucose. One `Glucose` biomarker with no fasting dimension means a normal post-meal 130 mg/dL flags `above` (see the separate entry below).

**The sharper form of the problem, and the reason this is not a curation detail:** HDL alone wants **three** ranges — male-normal, female-normal, and optimal — and *optimal* is a different axis from *sex*, not a third sibling on the same one (a sex-specific optimum is perfectly coherent). So the dimensionality is at least two-deep, and the axes are not interchangeable. HDL was left unseeded in WI-3 rather than resolve this under time pressure; **hormone panels make it unavoidable** — testosterone, estradiol, FSH, and LH reference ranges are strongly sex-specific and often age- or menopausal-status-specific, and the catalog already carries all four.

Candidate mechanisms, none decided:
1. **Cohort columns on `framework_ranges`** (`sex`, `age_min`/`age_max`, `fasting`), extending the natural key. Expressive; complicates the point-in-time resolution rule ([ADR-0005](adr/0005-reference-range-frameworks.md)), which is currently provably single-valued — a property [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §3 leans on hard — and demands the *reader* know the owner's sex/age, which no read path does today.
2. **Sibling frameworks** (`nih_medlineplus_lipids_male` / `_female`). No schema change; multiplies frameworks combinatorially and makes "compare against MedlinePlus" ambiguous for a client.
3. **Split the biomarker** (`Glucose (fasting)`), which is right where the *measurement* differs rather than the population — the fasting case, not the sex case.
4. **A cohort/applicability table** the ranges reference. Most general; most machinery.

Note the platform stores what a mechanism would need: `lab_draws.fasting` exists, and owner sex/age would come from a profile concept that does not exist yet — worth noticing that (1) implies inventing one. Trigger: **the first hormone-panel range entry** (the point where omitting is no longer an option), or any second real instance of a sex-specific range the owner wants flagged. Until then the honest behaviour is `no_range` — say nothing rather than something misleading — and a range is one catalog import away whenever the owner accepts a given interpretation ([ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §6).

**Fasting-state glucose — one biomarker or several? ([ADR-0030](adr/0030-biomarker-identity.md), [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §5)**
The ADA thresholds seeded for `Glucose` are **fasting** plasma glucose (`FPG`), but the catalog has a single `Glucose` biomarker and the comparison never reads `lab_draws.fasting`, so a normal post-meal 130 mg/dL flags `above`. It errs conservative (a false "out of range", not a false "normal" — unlike the hypoglycemia floor, which was the dangerous direction and is fixed), so it is a wrong flag rather than a hazard. Owner's leaning (2026-07-17): make it a **distinct biomarker, `Glucose (fasting)`**, for known-fasted draws — the measurement, its reference range, and its clinical meaning all differ, which is a biomarker-identity distinction rather than a range dimension, and it gives CGM (Phase 7) somewhere natural to live instead of contorting the same row. Open: what plain `Glucose` then means (random/unknown), whether CGM is a third entity, whether the resolver should treat the names as aliases (it must not — they are different measurements), and whether existing `Glucose` rows need migrating. Trigger: WI-4's CLI (the first surface entering glucose by hand), or the first CGM adapter. Related to the cohort entry above, but the proposed mechanism differs — split the entity, not the range. **WI-4 shipped 2026-07-17 without folding this in** ([ADR-0059](adr/0059-cli-manual-entry-implementation-decisions.md) §5): the seeded ADA range is fasting plasma `Glucose` (FPG), so a fasting `Glucose` draw flags correctly; the split only matters once non-fasting/CGM input arrives, and the trigger is now live for that case.

**Physically one-sided quantities — a floor datum on `biomarkers` ([ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §3)**
Range comparison deliberately assumes nothing about a quantity's sign: a censored `<0.1` is the interval `(-inf, 0.1)`, so against an explicit `[0, 10]` target it is `indeterminate` rather than `in_range`. That is the fail-safe reading — the engine has no basis to invent a floor — but it is *wrong* for the many catalog biomarkers that are physically non-negative (concentrations, counts, enzyme activities), where the true answer is knowable. Owner's proposal (2026-07-17): store the fact rather than assume it. Candidate shape is a nullable **`biomarkers.physical_min REAL`** — `NULL` = "no known floor" (today's behaviour, unchanged, so the change is additive and safe by default), `0` = "cannot be negative" — which generalizes past a boolean to a non-zero floor if one ever appears, and mirrors the `range_low = NULL` idiom already in use. Then `<0.1` becomes `[0, 0.1)` and lands `in_range` correctly, while the engine still assumes nothing where the owner has not spoken. Open: the column vs. a boolean; which biomarkers get seeded `0`; whether an *upper* physical bound (e.g. percentages ≤ 100) is worth the symmetry; and whether import should validate stored values against it. Trigger: the first censored result whose `indeterminate` verdict is visibly wrong to the owner — most likely in WI-4, when real below-detection results start arriving. **WI-4 shipped 2026-07-17 without folding this in** ([ADR-0059](adr/0059-cli-manual-entry-implementation-decisions.md) §5): a below-detection `<0.1` flags `indeterminate` (conservative, never wrong-dangerous), so it stays deferred; the owner's revisit condition is the first such verdict that reads as visibly wrong during real entry.

**Independent verification of the tier-B molar masses ([molar-mass-provenance.md](molar-mass-provenance.md), [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §4)**
Twelve of the nineteen seeded `biomarkers.molar_mass` values are **tier B**: fetched from PubChem by a subagent and reconciled against a quoted clinical conversion factor, but not confirmed at source by a human or by the reviewing session. Seven are tier A (PubChem fetched live in-session). The full table, the sources, and the four counter-intuitive values (BUN 28.014 as urea-nitrogen not urea; Triglycerides/Folate assay-calibration proxies; Cortisol's adapted unit basis) are documented in [molar-mass-provenance.md](molar-mass-provenance.md) — that document is the record; this entry is the outstanding action.

Why it stays open rather than being closed as "checked": the tier-B cross-check is that the value reproduces the published factor, but the factor is essentially `10/MW`, so the check is near-tautological *if the factor was back-computed from the MW rather than quoted*. The evidence against that is a citation (`PMC11132569`) nobody has opened. Nothing is suspected wrong — the point is that "verified" and "reported as verified" are different claims, and a wrong molar mass is **silent**: it produces a plausible number and therefore a wrong range flag, which is the failure class [ADR-0005](adr/0005-reference-range-frameworks.md) exists to close.

Scope of the exposure is narrow today: a molar mass is only read when a result's unit and its range's unit differ by a mass↔substance factor, and every currently seeded range is stated in the biomarker's own `canonical_unit`. So no seeded comparison uses one yet. Trigger: **the first result imported in molar units** (mmol/L cholesterol or glucose from a non-US lab, a CGM feed in mmol/L), or WI-4's CLI making molar entry routine — either turns a documented gap into a live one. Cheapest closure is opening PMC11132569 and confirming the twelve factors against the table.

**Snapshot consistency of an enriched read ([ADR-0053](adr/0053-read-endpoint-surface-and-pagination.md), [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §1, [ADR-0037](adr/0037-core-service-concurrency-and-driver.md))**
`GET /v1/lab-results?framework=` performs four reads on one connection in autocommit — resolve the framework, fetch the page, look up the page's biomarkers, resolve their ranges — so each sees its own implicit transaction. A catalog import committing between them can mix pre- and post-import state within a single response: a range from the new state compared against a `canonical_unit`/`molar_mass` from the old. Raised by CodeRabbit on PR #28 and verified as real. It is deliberately not fixed there: this is a single-user local platform with one Core Service process, the interleaving window is milliseconds, and both writers are the owner — so the realistic outcome is one page reflecting a mid-flight import the owner just triggered, not a wrong answer anyone would act on. (The `RuntimeError` path the finding also cites is unreachable — it needs a framework *delete*, and no delete path exists.) The fix is to wrap each enriched read in one deferred read transaction, giving SQLite's WAL snapshot semantics a boundary to hold; the cost is transaction management in the read path, which currently has none, and it wants a concurrency regression test to be worth anything. Trigger: a second concurrent writer (Phase 4's job/event surface, where an import is no longer owner-initiated and no longer obviously mid-flight), or the first observed mixed-state read.

**UCUM validation at the import boundary ([ADR-0031](adr/0031-units-and-ucum.md), [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §6)**
No import path validates that a unit string is well-formed UCUM. `biomarkers.canonical_unit`, `lab_results.unit`, and — as of Phase 3 WI-3 — `framework_ranges.unit` are all accepted as free text; `imports.py` does not import `healthspan.units` at all. This is deliberate rather than overlooked: singling out one column would be inconsistent, and an unparseable unit is fail-safe rather than silently wrong — it surfaces as a named `error` flag on every comparison ([ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §2), never a bad number. The cost is that the error surfaces at read time, far from the import that caused it, and `units.is_valid_unit` already exists to close it. Open: whether to validate all three columns at the boundary (consistent, fail-early, and a new `imports.py` → `units.py` dependency that pulls the UCUM grammar into the import path), or keep the read-time `error` flag as the only signal. Trigger: WI-4's CLI (the first interactive surface where a fail-early message is worth more than a late flag), or the first real unit typo that costs debugging time. **WI-4 shipped 2026-07-17 without folding this in** ([ADR-0059](adr/0059-cli-manual-entry-implementation-decisions.md) §5): the CLI defaults each result's unit to the biomarker's `canonical_unit` and does not validate UCUM client-side (consistent with the import boundary); an unparseable unit still surfaces as a named `error` flag, so it stays deferred pending the first typo that costs debugging time. Also a candidate check for the consolidated **Database integrity & sanity check** audit below.

**`draw_utc` format validation at the import boundary and schema ([ADR-0005](adr/0005-reference-range-frameworks.md), [ADR-0053](adr/0053-read-endpoint-surface-and-pagination.md), [ADR-0059](adr/0059-cli-manual-entry-implementation-decisions.md) §2)**
`lab_draws.draw_utc` is stored as free `TEXT NOT NULL` with no format check at the import boundary (`imports._domain_errors` validates only `draw_tz_inferred`/`fasting`) or in the schema (migration 0001 has no CHECK). But the whole date-handling scheme keys on it being a well-formed **hyphenated `YYYY-MM-DD` prefix**: point-in-time range resolution slices `draw_utc[:10]` and compares it *lexically* against `framework_ranges.effective_date` ([ADR-0005](adr/0005-reference-range-frameworks.md) §2, `ranges.resolve_ranges`), and the `draw_from`/`draw_to` read filters compare lexically too ([ADR-0053](adr/0053-read-endpoint-surface-and-pagination.md)). Lexical comparison is correct *because* ISO-8601 `YYYY-MM-DD` sorts chronologically — but only if the value actually is that form. A basic-format ISO-8601 value (`20260115`, or the timestamp `20260115T083000Z` — valid ISO-8601, but its first ten characters are `20260115T0`) or any malformed date stored verbatim silently mis-resolves the range with no error. **This is the write-boundary sibling of the UCUM-at-import question above** — the same "validate free text at the boundary vs. rely on a downstream signal" theme — except a bad `draw_utc` is *silently wrong* (a mis-dated flag), not fail-safe like an unparseable unit (a named `error`), so it is the stronger case for closing.

WI-4 closed this at the **CLI** surface only ([ADR-0059](adr/0059-cli-manual-entry-implementation-decisions.md) §2, `_validate_draw_utc`); `POST /v1/import` and any future adapter can still store a malformed `draw_utc`. Two **complementary, not either-or** controls — layered exactly as the codebase already layers the ADR-0030 value model and the `molar_mass` guard ([ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §4): boundary validation for a collected, named error, and a schema CHECK as the can't-happen backstop:
1. **Import-boundary validation** in `imports._domain_errors`, mirroring the `framework_ranges.effective_date` check already there: a collected `422` naming the offending row *before any write*, with full semantic rigor (real calendar date, UTC-only timestamp). Covers only import paths.
2. **Schema shape-CHECK** on `draw_utc` (e.g. `GLOB '[0-9][0-9][0-9][0-9]-[0-9][0-9]-[0-9][0-9]*'`): a malformed value cannot exist regardless of which code path wrote it. SQLite CHECK can only enforce the **shape** (first ten chars are `dddd-dd-dd`), not calendar validity or the UTC designator — but the shape *is* exactly the substr-safety invariant the lexical comparison depends on, so it is a well-targeted backstop. Its cost is an opaque apply-time `IntegrityError` rather than a per-row message, which is why #1 carries the UX and #2 is only the backstop.

Neither subsumes the other; both are worth landing (belt-and-suspenders, same as the value model). Trigger: pair it with the UCUM-at-import work above (same import-boundary-validation theme), or the first malformed `draw_utc` from a non-CLI writer. Also a candidate check for the consolidated **Database integrity & sanity check** audit below.

**Additive `ALTER TABLE` migrations must recreate the affected `*_current` view ([data-model.md](data-model.md), [ADR-0027](adr/0027-audit-trail-and-corrections.md))**
The migration-0001 `*_current` views are `SELECT * FROM <table> WHERE superseded_by IS NULL`, and SQLite binds `SELECT *` to the base table's columns at `CREATE VIEW` time. A later migration that does `ALTER TABLE … ADD COLUMN` will therefore not surface the new column through the current-state view until the view is dropped and recreated. Convention to adopt when the first additive migration lands: any migration adding a column to a content table also `DROP VIEW`/`CREATE VIEW`s that table's `*_current` view (and a schema-integrity test asserts the view column set matches the base table). Surfaced by the 2026-07-11 end-of-Phase-1 holistic review.

**Trigger update (2026-07-16, Phase 3 WI-3).** Migration 0005 is the first `ALTER TABLE … ADD COLUMN` to ship (`biomarkers.molar_mass`, [ADR-0058](adr/0058-range-comparison-implementation-decisions.md) §4) — but the hazard did **not** materialize, because `biomarkers` is a catalog table: it has no `superseded_by` column and therefore no `biomarkers_current` view to go stale. The question stays open with a sharpened trigger: **the first `ALTER TABLE … ADD COLUMN` on a *content* table (one carrying `superseded_by`, and thus a `*_current` view)** — `lab_results`, `lab_draws`, `body_composition`, `cgm_readings`, `wearable_daily`, `events`, `interventions`, `intervention_dose_history`, `subjective_observations`, `clinical_documents`, `analyses`. The general schema-integrity test proposed above (every `*_current` view's column set matches its base table) remains the durable fix and is worth landing with — or before — that migration, since it converts a silent wrong-answer bug into a failing test. That same view/base-table column-set check is also a candidate for the consolidated **Database integrity & sanity check** audit below.

**Database integrity & sanity check — a deliberate, user-invoked audit ([ADR-0038](adr/0038-backup-execution-and-verification.md), [ADR-0027](adr/0027-audit-trail-and-corrections.md), [ADR-0035](adr/0035-migration-execution-semantics.md))**
There is no on-demand, report-only audit of the live database — a `chkdsk`-for-the-`.db` the owner can run deliberately. The primitives exist (`db.integrity_check` / `db.foreign_key_check`, wrapping `PRAGMA integrity_check` and `PRAGMA foreign_key_check`) but only as **gates inside other operations**: backup/restore verification ([ADR-0038](adr/0038-backup-execution-and-verification.md)'s "verified" definition), the migration runner's pre-commit `foreign_key_check` ([ADR-0035](adr/0035-migration-execution-semantics.md)), and the pre-rekey verified backup ([ADR-0028](adr/0028-key-derivation-and-rotation.md)). Startup `service.verify_schema` is narrower still (`schema_version` + the reserved `not_assigned` row). The `db` CLI exposes `migrate`/`backup`/`restore` — no `check`.

The concept is a deliberate `healthspan db check` that audits the whole database and **reports** (never silently gates, never auto-repairs), read-only by default — any future `--fix` a separate, backup-first escalation, since repairing a health database is the irreversible-action class the platform makes the owner confirm. Two layers:
1. **Physical + referential** — run [ADR-0038](adr/0038-backup-execution-and-verification.md)'s already-defined verification (`integrity_check` + `foreign_key_check` + `schema_version`) against the *live* database on demand, rather than only when producing a backup. High reuse; the machinery exists.
2. **Application / domain invariants ("sanity")** — the promises **no `CHECK` or foreign key can express**, which are therefore *silently wrong* if violated: `draw_utc`/`effective_date` format (the entries above), UCUM unit validity (above), `*_current` view column-sets matching their base tables (above), supersession-chain consistency (exactly one current row per natural key, `superseded_by` → a real newer row, no cycles — [ADR-0027](adr/0027-audit-trail-and-corrections.md)), `alias_normalized` disjoint from normalized `canonical_name` ([ADR-0054](adr/0054-biomarker-name-alias-fallback.md)), the reserved `not_assigned` row ([ADR-0055](adr/0055-biomarker-category-taxonomy.md)), audit-log coverage. This is the **consolidation point** for those scattered invariant items — they stop being one-off deferrals and become the initial check suite of one deliverable. Until that suite is scoped, the standalone entries above remain the authoritative deferral records: each keeps its own trigger and may resolve independently (e.g. as write-path validation) before `db check` exists — this entry consolidates them, it does not supersede them.

Why it earns its place even alongside write-time validation and schema CHECKs (the belt-and-suspenders of the `draw_utc` entry above): an on-demand audit is the only thing that catches a row written *before* a constraint existed, a migration bug, a direct-database edit, or bit-rot — failures no write-path guard can see. Architecturally it is a new command + subsystem (a new `db check` surface, a new invariant-audit module), so it warrants its **own Proposed ADR when scoped**; this entry is the lightweight capture until then. Trigger: when the subsumed invariant items above accumulate enough to warrant building the suite rather than closing them one at a time, or the first real integrity/invariant-violation incident, or the post-Phase-3 real-data-accumulation milestone where "is my data actually consistent?" becomes a question the owner asks.

**Derived data points — deferred schema ([ADR-0044](adr/0044-derived-data-points.md))**
ADR-0044 fixes the principle (derived values are a distinct content class, never stored in source-data tables; interim home is the `result_data` attachment on analyses) and deliberately defers the derived-series table. Sub-questions recorded for when it is designed: (1) internally computed (formula stored, rebuildable) vs. externally computed (opaque snapshot) — the two subtypes behave differently under the [ADR-0027](adr/0027-audit-trail-and-corrections.md) correction model; (2) does a derived series get a `biomarker_id` ([ADR-0030](adr/0030-biomarker-identity.md))? (3) may derived series be range-compared ([ADR-0005](adr/0005-reference-range-frameworks.md)) — HOMA-IR has published ranges, so a blanket "never" is wrong; (4) staleness marking when a source row feeding a derived point is superseded. Design trigger: the analyses table exists in a running system and accumulated attachments show which subtype dominates.

**Accumulation evidence (2026-07-14, Phase 3 D1 discussion).** Two concrete *externally-computed* instances from the owner's Function Health data, recorded to inform sub-questions (1) and (3): **Biological Age** (opaque composite, unit years) and the IGF-1 **Z-Score** (opaque, unit SD, **published reference range −2 to +2**). Both are values the owner cannot recompute but wants imported for longitudinal analysis. They sharpen the deferral in two ways: (1) the externally-computed subtype behaves like *received source data* under the correction model — correcting the owner's stored IGF-1 does not invalidate a Z-Score Function computed from its own measurement, so the "never in source tables" hazard (rebuildable staleness) does not apply to it the way it does to internally-computed HOMA-IR; (3) the Z-Score's −2..+2 range is a live counterexample to any blanket "derived values are never range-compared." These do **not** resolve the deferral (still Phase-5-triggered) and are explicitly **not** Phase 3 biomarkers — importing them is ADR-0044-gated, not part of the [ADR-0055](adr/0055-biomarker-category-taxonomy.md) taxonomy.

**Recovery Kit OS print pathways ([ADR-0033](adr/0033-plaintext-artifact-disposal.md), [ADR-0047](adr/0047-crypto-surface-implementation-decisions.md))**
WI-2 shipped kit rendering as terminal display plus explicit `--output` file (ADR-0047 §4). The **orphan startup sweep shipped in Phase 2 WI-1** (`recovery_kit.sweep_orphans`, called from Core Service startup; [ADR-0049](adr/0049-core-service-skeleton-implementation-decisions.md)), best-effort (overwrite-then-unlink). **Sweep target narrowed 2026-07-12 (full-codebase review finding):** it originally globbed `*recovery-kit*`, which also matched the deliberate `--output` kit filename — so a user who saved their kit into the data directory would have their only offline key copy silently disposed on the next `service start`. The sweep now targets only the disjoint spool naming `recovery_kit.ORPHAN_SPOOL_GLOB` (`.healthspan-recovery-kit-*.spool`), which a deliberate `.txt` kit can never match. Still deferred: ADR-0033's OS print integration (`lp`/`lpr` streaming, Windows temp-file shell print with verified disposal) — the sweep's only *producer* of orphans, **and it must write its transient file via `recovery_kit.orphan_spool_filename` so the sweep can catch a crash between render and disposal**. The print pathways land with the user-documentation milestone; until then the sweep hook has no producer. Trigger: user-documentation milestone (or the first Windows shell-print pathway).

**Direct-database exclusive-access enforcement ([ADR-0038](adr/0038-backup-execution-and-verification.md), [ADR-0042](adr/0042-process-supervision-and-single-instance-locking.md))**
Phase 2 WI-1 shipped the ADR-0042 single-instance advisory lock (`healthspan/locking.py`) and applied it as `cli_support.exclusive_database_access` to the three sanctioned direct-database commands — `db migrate`, `db backup`, `db restore` — which now acquire `<database-path>.lock` fail-fast and refuse while the Core Service (or another instance) holds it ([ADR-0049](adr/0049-core-service-skeleton-implementation-decisions.md) §6). `db restore` holds it for its whole duration, per ADR-0042. The `keys` rekeying commands (`change-passphrase`/`rotate-secret-key`/`convert-mode`) joined the guard in **WI-2b** (the deferred item (1)): each acquires the lock fail-fast before its first prompt and holds it through the rekey. **Still open:** the in-service scheduled `backup.database` job (ADR-0038's scheduled producer, of which the WI-4 CLI is the offline counterpart) — deferred until the job/scheduler surface exists.

**Keyring credential entries are machine-global — multi-database operation ([ADR-0026](adr/0026-named-scoped-tokens.md), [ADR-0051](adr/0051-auth-lifecycle-and-rate-limiting-implementation-decisions.md))**
ADR-0026 fixes the keyring convention as service `healthspan`, usernames `token:<name>` and `mcp-client-secret` — one namespace per machine. The WI-2b bootstrap therefore overwrites those entries whenever it mints into *any* empty `tokens` table: starting the service against a second database (a test database, a fresh `--config`) silently replaces the first database's stored plaintexts, whose hashes remain only in the first database — the CLI then presents the wrong credential and gets uniform 401s. Single-database operation is the design center and unaffected. Resolving multi-database support would need per-database namespacing of the keyring entries, which changes an Accepted ADR-0026 convention → extension ADR. Trigger: any decision to support more than one live database per machine (or the first user report of the clobber).

**Non-loopback binding hardening ([ADR-0049](adr/0049-core-service-skeleton-implementation-decisions.md), [security.md](security.md) Network Security)**
Phase 2 WI-1 added the `[service] host` config key (default `127.0.0.1`, loopback-only). The parser *accepts* a non-loopback bind (`0.0.0.0` or a specific interface), but security.md's Network Security controls that a LAN bind requires — Host-header validation and CORS as DNS-rebinding defenses, and HTTPS for any non-loopback address — are **not implemented** in `service.py`/`_run_uvicorn`. Impact is currently bounded: the only live route is the status-word-only unauthenticated liveness endpoint. But a user who repoints `host` and later gains authenticated data routes would be exposed, so **non-loopback binding is unsupported until these controls land**. Trigger: the first authenticated data route (WI-2/WI-3), where the controls belong with the auth/middleware layer — or an explicit LAN-deployment milestone.

**Broadened by WI-4 (2026-07-17, PR #31, CodeRabbit).** The risk now also has a **client-side / outbound** face: WI-4's CLI (`cli_client.default_client`) sends `Authorization: Bearer` requests to `service.host` over `http://`, so a non-loopback `service.host` puts the token on the wire in cleartext (CWE-319) independent of what the *server* binds to — and `config._parse_service` accepts any non-empty host ([ADR-0049](adr/0049-core-service-skeleton-implementation-decisions.md) §Binding posture makes a LAN bind a deliberate, user-typed opt-in), so nothing rejects it. A related sub-vector was found and **closed** in the same PR: `default_client` sets `trust_env=False`, so an inherited `HTTP_PROXY`/`HTTPS_PROXY` cannot route even the *loopback* bearer request through an external proxy (httpx trusts those by default). The remaining deferral is therefore specifically the non-loopback-**host** controls, not the proxy angle. Candidate resolutions when taken up: **(a)** implement security.md's controls (Host-header validation, CORS, HTTPS) and keep ADR-0049's explicit LAN opt-in — honors both, no revisit; or **(b)** reject a non-loopback `service.host` in `_parse_service` (CodeRabbit's suggestion), which reverses that ADR-0049 opt-in decision — an amendment while ADR-0049 is Proposed, a superseding ADR once it is Accepted. Either is a legitimate Phase-3 action; both sit outside WI-4's client scope. **Trigger (sharpened):** resolve before the platform supports any non-loopback *authenticated* deployment — a Core Service bound beyond loopback, or a CLI `service.host` pointed at a remote host — since the parent trigger (first authenticated data route) has already fired.

**Authenticated API-docs exposure ([ADR-0049](adr/0049-core-service-skeleton-implementation-decisions.md) §7, [ADR-0040](adr/0040-health-endpoint-authentication.md))**
The Core Service app is built with `openapi_url=None`, `docs_url=None`, `redoc_url=None` — FastAPI's auto-generated `/openapi.json`, Swagger, and ReDoc are off ([ADR-0049](adr/0049-core-service-skeleton-implementation-decisions.md) §7): they would be additional unauthenticated routes (breaking the exactly-one-`public`-route property), and an unauthenticated schema is fingerprinting material ([ADR-0040](adr/0040-health-endpoint-authentication.md)'s "minimize what an unauthenticated caller can learn"). Undecided: whether and how to expose them **behind auth** (e.g. `monitor`- or `admin`-gated) rather than not at all. [api-reference.md](api-reference.md) remains the design-time reference meanwhile — the practical need for a *live* schema has not yet arisen. Trigger: the first client-development need for machine-readable OpenAPI (most plausibly Phase 4's MCP work or the Phase 6 GUI), or an explicit owner request.

**`psutil` PID-identity refinement for the single-instance lock sentinel ([ADR-0042](adr/0042-process-supervision-and-single-instance-locking.md), [ADR-0049](adr/0049-core-service-skeleton-implementation-decisions.md) §6)**
The kernel advisory lock is the correctness guarantee; the lock sentinel records the holder PID only for a human-readable "held by" message, so a reused PID can make that message name the wrong process — cosmetic, never a correctness hazard. [ADR-0042](adr/0042-process-supervision-and-single-instance-locking.md) specifies the `psutil` create-time hygiene check as the refinement; Phase 2 WI-1 deliberately did not pull it in ([ADR-0049](adr/0049-core-service-skeleton-implementation-decisions.md) §6). Trigger: the supervision work (Phase 6), where `psutil` and process-identity handling arrive anyway.

**Rekey crash-durability barrier ([ADR-0028](adr/0028-key-derivation-and-rotation.md), [ADR-0047](adr/0047-crypto-surface-implementation-decisions.md))**
`_rekey` preserves the `.pending` sidecar once `db.rekey` has succeeded, so an interrupted rotation is *recoverable* — `unlock()` detects the stray `.pending` and guides the fix (ADR-0047 §7). Full crash-*consistency* — an `fsync` of the rekeyed database pages before, and of the directory after, the `os.replace(pending, sidecar)`, under `synchronous=NORMAL` WAL — is not yet enforced, so a power loss in the narrow window between the sidecar install and the pages reaching disk could still require the pending-file recovery rather than opening cleanly. Deferred: a correct barrier needs crash-injection testing and is platform-nuanced. Trigger: the process-supervision / durability hardening pass (Phase 2+, alongside the ADR-0042 lock). Surfaced by the 2026-07-11 end-of-Phase-1 holistic review.

- **Biomarker molar-mass persistence** → A nullable `biomarkers.molar_mass REAL CHECK (molar_mass IS NULL OR molar_mass > 0)` column (migration 0005, Phase 3 WI-3), added by a plain `ALTER TABLE … ADD COLUMN` — not a rebuild, so the `legacy_alter_table` FK-repointing hazard migration 0004 documents never arises. The catalog is the natural home: molar mass is per-biomarker reference data sitting beside `canonical_unit`, the other column the comparison path reads for the same purpose. Nineteen biomarkers seeded from PubChem, each cross-checked against an independently published clinical conversion factor; three carry recorded subtleties (BUN is the **urea-nitrogen equivalent** 28.014, not urea's 60.06 — BUN measures nitrogen mass and each urea carries two N, so urea's own mass cancels out of the conventional 0.357 factor; Triglycerides 885.4 and Folate 441.4 are assay-calibration proxies, not the physiologically dominant species). Primary axis is **physiological system**; a theme earns a category only when its members have no system home (`environmental_toxins`, `screening`), else it is a future tag. Single primary category per biomarker now; cross-cutting tags deferred additively (see the open "Biomarker cross-cutting tags" entry above). Nineteen-category system-axis seed (`autoimmunity`, `allergy`, `body_composition`, `electrolytes`, `environmental_toxins`, `heart`, `hematology`, `hormones`, `immune`, `inflammation`, `kidney`, `liver`, `lipoproteins`, `metabolic`, `nutrients`, `pancreas`, `screening`, `thyroid`, `urine`) + the reserved default, owner-editable data forever after. Computed scores (Biological Age, IGF-1 Z-Score) are ADR-0044 derived data, not categorized biomarkers. Decided 2026-07-14 (Phase 3 decision gate D1). See [ADR-0055](adr/0055-biomarker-category-taxonomy.md).

- **Audit trail** → `audit_log` table in migration 0001: append-only (immutability enforced by schema triggers), written by the Core Service data-access layer in the *same transaction* as every mutation so it cannot drift. - **Longitudinal data correction** → `superseded_by` self-FK on every data table (option a). Value corrections insert the corrected row and point the original at it — never mutate in place; current state is `WHERE superseded_by IS NULL` via per-table `*_current` views. Carve-out: designated metadata repairs (the timezone correction workflow) update in place, fully audited. Deletes are hard deletes with a mandatory audit row preserving the full row image; clients flag the request and offer a backup first; supersession-chain rows are not deletable. Writes exclusively via the validated Core REST path (mutation + audit row in one transaction, Core-emitted `data.*` event); reads hit current-state tables/views or ADR-0021 aggregates, which are rebuildable caches — never authoritative — invalidated by `data.imported`/`data.corrected`/`data.deleted` events. - **Timezone storage convention** → UTC as ground truth (ISO 8601). Every timestamp table carries four columns: `*_utc` (UTC, ground truth), `*_local_recorded` (original value from source, immutable), `*_local_tz` (IANA timezone name, best guess), `*_tz_inferred` (boolean — 1 if timezone was assumed not known). Correction workflow: update `local_tz`, recompute `*_utc` from `local_recorded`, clear `tz_inferred`. See [design-rationale.md](design-rationale.md) for the full convention.

- **Intervention dose history** → `intervention_dose_history` child table with FK to `interventions`. Key fields: `change_type` enum (`initiation`, `increase`, `decrease`, `hold`, `resumption`, `discontinuation`), `authority_type` enum (`prescribing_physician`, `supervising_clinician`, `self`, `protocol`), `ordered_by` free text (NULL when self), `reason` enum (`scheduled_titration`, `lab_result`, `symptom_response`, `side_effect`, `cost_or_availability`, `physician_directed`, `protocol_change`, `other`). `authority_type` and `reason` are orthogonal — a lab-result-driven change can be either physician-directed or self-adjusted. See [data-model.md](data-model.md).

- **Cloud backup strategy** → Cloud backup of the encrypted SQLite file is explicitly safe and recommended — but only the output of `healthspan db backup`, never the live database file. The database is AES-256 ciphertext (SQLCipher, ADR-0013), so the cloud provider cannot read it, and the provider is in the "do not trust" tier for storage — the encryption model handles confidentiality correctly. That is a separate question from consistency: the live file runs in WAL mode, so a sync client snapshotting it mid-write can capture a torn, unrecoverable copy — encryption makes a torn copy worse, not better, since it can't be partially salvaged. `healthspan db backup` produces a checkpointed, self-consistent, encrypted snapshot; that snapshot — and only that snapshot — is safe to sync continuously to Dropbox, iCloud Drive, OneDrive, Backblaze, or any similar service. See [ADR-0019](adr/0019-multi-device-sync.md) for the single-writer + backup-only-sync pattern.

- **Process lifecycle** → Launcher script default; Docker Compose supported. See [ADR-0008](adr/0008-process-lifecycle.md).

- **Clinical documents — full-text search strategy** → SQLite FTS5 external-content virtual table over the `body` column, tokenizer `porter unicode61 remove_diacritics 2` (recall-favoring for AI natural-language search; rebuild-reversible). The index inherits the SQLCipher boundary — its shadow tables are ordinary encrypted SQLite tables — so there is no second security surface. Sync is by INSERT/UPDATE/DELETE triggers shipped in migration 0001 with the documents table; the index covers all rows and current-state is filtered at query time via a `superseded_by IS NULL` join, keeping the triggers mechanical under the ADR-0027 correction model. Application-level scan is rejected (dead end as narrative accumulates); embedding/semantic search is a future additive plugin layer over FTS ([ADR-0010](adr/0010-cli-plugin-model.md)), not a replacement, and its vector store must live inside the encryption boundary. See [ADR-0041](adr/0041-clinical-document-fts.md).
