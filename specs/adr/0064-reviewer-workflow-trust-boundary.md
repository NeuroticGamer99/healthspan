# ADR-0064: Reviewer Workflows Execute Only Default-Branch Code (extends ADR-0045)

## Status
Proposed

## Context and Problem Statement
The Antigravity (Gemini) reviewer — [`gemini-review.yml`](../../.github/workflows/gemini-review.yml) plus [`gemini_review_agent.py`](../../.github/scripts/gemini_review_agent.py) and its tested logic module — is the repository's first CI job that holds a **third-party API key** (`GEMINI_API_KEY`) *and* a **`pull-requests: write`** token, and whose entire subject matter is untrusted input: a pull request's diff.

As first landed, the job checked out `main`, then ran `gh pr checkout "$PR"` — swapping the worktree to the pull-request head — and executed the agent from there. Three executable artifacts therefore resolved from PR-controlled files: the agent script, the `scripts/gemini_review_logic.py` it imports, and the script's **PEP 723 `dependencies:` block**, from which `uv` installs before a line of agent logic runs. Changing that one line to name a hostile package is arbitrary code execution beside both credentials. Both CodeRabbit and Copilot flagged it independently on the introducing PR; it was deferred to a focused change ([issue #58](https://github.com/NeuroticGamer99/healthspan/issues/58)).

Practical exposure was low — a solo repository, manual `workflow_dispatch`, no external contributors — but the shape is not self-limiting. The "fork pull requests skip the reviewer" property that protects `pull_request`-triggered jobs does **not** apply here: `workflow_dispatch` plus `gh pr checkout` will happily run a fork's code, so the day a contributor arrives, a maintainer dispatching a review on their pull request hands it the key. The reviewer already invests heavily in containment (sensitive-path exclusions mirrored from `.coderabbit.yaml`, a deny-all agent tool policy, a pre-tool hook that walks nested tool arguments) — all of which an attacker skips by editing the dependency line instead.

What no document decided is the **general rule** this instance needs: which code a credential-holding workflow may execute, and what a pull request is allowed to be to it.

This ADR **extends [ADR-0045](0045-repository-workflow-and-ci-enforcement.md)** — whose §6 and §7 cover what CI runs and how its tooling stays pinned, and whose supply-chain discipline (SHA-pinned actions, hash-verified binaries) this makes explicit for the *code under review* — and reverses none of its decisions.

## Decision Drivers
- The credentials are intrinsic to the feature, not incidental: the agent authenticates to Gemini and `POST`s a real review to `/repos/{owner}/{repo}/pulls/{n}/reviews`. Dropping either is not a fix; confining what runs beside them is.
- A trust boundary must hold by construction, not by a reviewer noticing. `gh pr checkout` reads as innocuous plumbing — it is one line, it is the documented way to get a PR locally, and it silently redefines *every* subsequent path in the job.
- The reviewer is only useful if it reviews the pull request's actual diff, so "just don't fetch the PR" is not available. Data and code have to be separated, not the PR excluded.
- `workflow_dispatch` gives the job **no** inherited trust: the workflow file that runs is the one on the **dispatched ref**, which need only share a filename with one on the default branch to be dispatchable. A job that wants its executed artifacts pinned to the default branch has to pin them itself — nothing about the event does it.
- `scripts/bot_review.py`'s run attribution — the `run-name: "Gemini review: PR N"` title match (`run_is_for_pr`) and the `--run <id>` fail-fast — is load-bearing for the whole request/wait/fetch chain and must survive untouched.
- Whatever is chosen has to be believable without a live run: the SDK glue is CI-only and has never executed end-to-end (this hardening merges *before* the reviewer's first real use), so a mechanism whose safety depends on unobserved SDK behavior cannot be assessed yet.

## Considered Options
1. **Pin execution to the default branch, fetch the PR head as data** (chosen) vs. **keep `gh pr checkout` and drop the credentials** vs. **keep it and gate dispatch on a maintainer allow-list**. Dropping credentials deletes the feature (no key, no review; no write token, nowhere to post). An allow-list is a *procedural* control over a *structural* hole — it still executes the PR's code, just after a human decided the PR looked fine, which is precisely the judgment a hostile dependency line is designed to survive.
2. **`git fetch` the PR head into a ref of our own** (chosen) vs. **`FETCH_HEAD`** vs. **resolving the SHA through `gh api` and fetching by object name**. A named ref anchors the objects and stays re-readable; `FETCH_HEAD` is clobbered by any later fetch. Fetching a bare SHA depends on the server allowing reachable-SHA1 requests and does not resolve a fork's head at all, while `refs/pull/N/head` does.
3. **`view_file` keeps reading the `main` worktree, with the instructions reframed** (chosen) vs. **materializing PR blobs into a read-only path** vs. **denying `view_file` entirely**. Discussed in Decision Outcome §3.
4. **Widen the diff's unified context lines to compensate** — rejected for now: it inflates the prompt against a free-tier quota and widens the anchorable-line set, and there is no evidence yet that context is the binding constraint. A one-line change once a real run says otherwise.

## Decision Outcome

### 1. Credential-holding workflows execute only default-branch code; pull requests are data
The rule, stated generally because the next reviewer or CI job will face it too:

> A workflow step that holds a secret or a write-scoped token executes only code resolved from the repository's default branch. Pull-request content may be **read** by that step — as a diff, as fetched git objects, as text in a prompt — but never **executed**: not as a script, not as an imported module, not as an installed dependency.

Concretely, in [`gemini-review.yml`](../../.github/workflows/gemini-review.yml): `actions/checkout` pins `ref: ${{ github.event.repository.default_branch }}` and the worktree **stays** there. The pin is the anchor of the whole boundary, not tidiness — unpinned, the checkout follows the dispatched ref, and a review dispatched on any other branch would execute *that* branch's agent script and PEP 723 dependencies beside both credentials. `gh pr checkout` is gone, replaced by `git fetch --no-tags origin "+refs/pull/$PR/head:refs/gemini-review/pr-head"` — which moves objects into `.git` and executes nothing. `refs/pull/N/head` is used because it resolves a fork's head as well as a branch's, and the SHA is read back with `git rev-parse --verify ...^{commit}` in its own assignment, so a fetch that produced no usable ref fails *this* step rather than publishing a ref name as the SHA. The fetch step needs no token: the checkout's persisted credentials authenticate it, and `persist-credentials: true` is stated on the checkout rather than inherited so the dependency is visible in the file. The head SHA is published as a step output and passed to the agent as `PR_HEAD_SHA`.

### 2. The reviewed range names the head SHA, and refuses anything that is not one
`filtered_diff` previously diffed `origin/main...HEAD`, correct only because the worktree *was* the PR. With the worktree pinned, `HEAD` is `main`: unchanged, that code reviews an empty diff and posts a **clean review of a pull request it never looked at** — a silent false negative, the worst available failure. The range is now `origin/main...$PR_HEAD_SHA`, built by `diff_argv` in [`gemini_review_logic.py`](../../scripts/gemini_review_logic.py) — the tested, pyright-checked module, not the CI-only script — which **rejects any `head_sha` that is not a bare 40- or 64-hex commit SHA**. That check is fail-closed against two silent-wrongness modes: a ref name would review the wrong commits, and a value beginning with `-` would be read by git as an option. Both would otherwise be reported as an ordinary review. `tests/test_gemini_review_logic.py` proves the range end to end against a real git repository — worktree on `main`, head reachable only as a fetched object, PR content still in the diff — and asserts the `HEAD` form is empty in exactly that shape.

The base stays the literal `origin/main`, matching `ci.yml`'s `branches: [main]` and the `origin/main` in four `.claude/skills/` files: a default-branch rename is a repo-wide edit event, not something one line should paper over. But §1's checkout resolves the default branch *dynamically*, and those two can diverge silently in one shape — the default branch redesignated while a stale `main` survives, so the job executes the right code and diffs the wrong base, posting a review of unrelated changes as an ordinary one. (With no `main` left at all, `git diff` exits non-zero and the run fails loudly already.) `verify_diff_base` closes it: the workflow passes `github.event.repository.default_branch` to the agent as **data only** — never as an argument to git, which would put a second caller-supplied string in a security-critical argv — and the agent refuses to run when it disagrees with `DIFF_BASE`, naming both values and what to edit. Raised by Copilot on the implementing pull request; its own suggested remedies were declined (`refs/remotes/origin/HEAD` does not exist under `actions/checkout`, which never runs `git remote set-head`; threading the base into `diff_argv` reverses this ADR's single-validated-input property).

### 3. `view_file` reads `main`: base-version context, declared to the model
The agent's tool policy allows `view_file`, which reads the worktree — now `main`, so for any file the diff touches it returns the version **before** the change. Two alternatives were weighed. Denying `view_file` outright removes the stale-read hazard but also the reviewer's ability to check a caller or a helper's signature, which is where a diff-only reviewer produces its worst false positives. Materializing the PR's blobs into a read-only path gives the richest context, but it builds a new containment guarantee — "read from here, never execute from here" — on `view_file` path semantics this repository has never observed, inside the very change meant to shrink the trust surface.

Chosen: keep `view_file`, and make the model's mental model correct instead. The system instructions now state that `view_file` reads the default branch and **not** the pull request, that touched files are therefore pre-change on disk, that the reads are before-and-after context only, and that no finding may rest on a changed file's on-disk content — the diff is the sole authority on what the PR does. This costs nothing in trust surface and is honest about what the tool returns. The degradation is also narrower than it first appears: reads of *unchanged* files (most context lookups) are identical to the PR's, and for changed ones the base version plus the diff is the whole picture. Option 2 remains available on evidence from real runs.

### 4. `pull-requests: write` stays; the residual is prompt injection, and it is accepted
The write scope is what makes the reviewer a reviewer. With the surrounding Python trusted, the remaining attack is the model itself: diff content crafted to steer the review. Its blast radius is an **unfaithful review** — the agent's tools are deny-all except `view_file` and `finish`, its output is a structured `ReviewResult`, and the token is never in the model's reach — so the worst outcome is a misleading finding, or a real one suppressed. That is bounded by the standing rule that every bot finding is verified against the code before it is acted on ([`.claude/bot-review-triage.md`](../../.claude/bot-review-triage.md)), and it is noted in the agent's docstring. Accepted, not closed.

### 5. The prohibition is asserted by a test, not left to review
`tests/test_gemini_review_logic.py` reads the workflow source and fails on any worktree-swapping construct (`gh pr checkout`, `git checkout`, `git switch`, `git worktree`, `git reset`) outside a comment, on a checkout step that does not pin the default-branch `ref:`, and on any use of the PR ref by a verb other than `git fetch`. The comments and this ADR explain *why*; the test is what makes the boundary fail the build. It follows the `EXCLUDED_GLOBS`/`.coderabbit.yaml` mirror test already in that file — the repository's established answer to "a containment boundary that only a reviewer's attention enforces" — and needs no new CI job or dependency: it hand-scans text, as that check does, since the project declares no YAML parser.

### 6. The reviewer cannot review its own hardening
The mechanized ask dispatches `ref=<default branch>` (`bot_review.py::_request_by_dispatch`), so a review of this pull request would run **`main`'s** copy of the workflow — pre-merge, the version *without* this hardening: the vulnerable path, pointed at the pull request about the vulnerability. This change is therefore reviewed by the other chains (CodeRabbit, Copilot), and the reviewer's first live end-to-end run happens on the first pull request *after* it merges.

The `ref` is the dispatcher's choice, not a property of the event (Decision Drivers). Dispatching some other ref runs that ref's copy of this file, whose `permissions:` block and step order that ref controls — §1's checkout pin cannot bind a workflow file that was itself replaced. That residual is bounded by repository write access, since only a ref pushed to this repository can be dispatched (a fork's branch cannot): anyone who can create it can already run arbitrary code in Actions by other means. It is out of this ADR's reach and named here so it is not mistaken for something the pin closed.

### Positive Consequences
- The credentials' blast radius no longer depends on what a pull request contains. The agent script, its imported module, and the PEP 723 dependency set are all `main`'s, so the containment already built (path exclusions, deny-all policy, the nested-argument hook) can no longer be stepped around by editing a dependency line.
- The reviewer becomes safe to point at a fork or contributor pull request — the case it would silently have failed.
- The silent-clean-review mode this change could itself have introduced — worktree on `main`, so `HEAD` diffs to nothing — is closed twice: by naming the head SHA, and by a fail-closed check that refuses every non-SHA value rather than diffing something plausible. (It does not close the pre-existing empty-range case; see Negative Consequences.)
- The security-critical range construction moved from the untested CI-only script into the pyright-checked, unit-tested logic module, matching the split that module exists for.
- The boundary is enforced by a test (§5), so undoing it reddens the build instead of requiring a reviewer to catch one line.
- `bot_review.py` is untouched: `run-name`, the dispatch ask channel, and the `--run` fail-fast all work exactly as before.

### Negative Consequences / Tradeoffs
- `view_file` returns pre-change content for changed files. Mitigated by instruction, not by mechanism — a model that ignores the instruction can still produce a finding grounded in stale content. Accepted as the cheapest fully-safe option; §3's option 2 is the escalation if real runs show it matters.
- The §5 test scans workflow *text*, so it recognizes the constructs it lists and a checkout step shaped like today's. A future GitHub feature that reaches the same end differently — or a rewrite that restructures the step — passes it. It raises the cost of undoing the boundary from "nobody notices" to "delete a named assertion"; it is not a proof.
- Prompt injection of the model remains open (§4), bounded but unclosed.
- An empty filtered diff still posts a review stating 0 findings, so a dispatch against an already-merged pull request (its head an ancestor of `main`) reads to `bot_review.py`'s count regex exactly like a clean review. The body says "The filtered diff is empty — nothing to review" and `fetch` prints it, so it is visible rather than silent — but only to a reader, not to the count cross-check. This predates the change and is untouched by it: distinguishing the two states means a new marker and a `bot_review.py` change, which this security fix deliberately keeps out of scope.
- The end-to-end path stays unexercised until the next pull request merges after this one; the tests cover the range logic against real git, not the assembled job.
